A traditional keyword-based database search would not be sufficient for searching 500-page contracts via natural-language questions like “What are the termination clauses?”. Keyword search relies on exact or near-exact matches, so it can miss relevant sections when the contract uses different phrasing (e.g., “termination”, “cancellation”, “expiry”, “early exit”, “material breach”, “notice period”) or when the lawyer asks a conceptual question (e.g., “How can either party end the agreement?”). It also tends to return many false positives because common legal terms appear throughout a contract, forcing users to manually sift results.

A vector database would support semantic search by storing embeddings of contract text (typically split into chunks such as paragraphs or sections). When a lawyer asks a question, the system embeds the query and retrieves the most semantically similar chunks, even if the exact words do not match. This improves recall and relevance, and it enables question-style queries rather than keyword-style queries. In practice, a vector database would sit behind a retrieval-augmented generation (RAG) workflow: retrieve the most relevant clauses (via vector similarity), then optionally feed those passages into an LLM to produce a concise answer with citations. Keyword search can still be useful as a secondary filter, but vectors are key for meaning-based retrieval.
